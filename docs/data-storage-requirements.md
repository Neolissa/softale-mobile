# Системные требования к хранению ключевых данных

Этот документ фиксирует обязательные требования к сохранению аккаунта, профиля и прогресса в SofTale (web/mobile, server mode).

## 1) Ключевые данные (обязательны к сохранению)

- Идентичность:
  - email пользователя (уникальный ключ),
  - роль (`USER` / `ADMIN`),
  - пароль в виде хеша (никогда не в открытом виде).
- Профиль:
  - `displayName`, `aboutMe`, `gender`, `isAdult18Plus`,
  - `avatarUri` (постоянный URL, не `blob:`).
- Игровой прогресс:
  - `selectedStory`, `selectedCourseId`, `selectedDifficulty`,
  - `startedStoryIds`, `completedCount`,
  - `unlockedEndings`, `unlockedAchievements`,
  - `questRatingStats`, `practiceStats`.
- Диагностика и результат теста:
  - `diagnosticCompleted`,
  - `conflictPrimaryStyle`, `conflictSecondaryStyles`.
- Экономика:
  - `wallet` (`xp`, `energy`, `level`) как источник истины,
  - связанная копия в `profile` должна быть синхронизирована с `wallet`.
- Сервисные данные:
  - `lastSeenAt`, `claimedDailyEnergyAt`,
  - `grantedPerfectStageIds`, `redeemedPromoCodes`,
  - `eventProgress`, `unlockedPaidStageKeys`.

## 2) Требования к надежности хранения

- Все ключевые данные хранятся на backend в `DATA_DIR` (production: `/var/data`).
- База (`db.json`) должна жить на persistent disk (не эфемерный контейнерный FS).
- На старте сервера обязателен автобэкап БД в `DATA_DIR/backups` с ротацией.
- Обновления приложения/фронтенда не должны стирать БД.
- Обновления backend не должны менять `DATA_DIR` и не должны отцеплять диск.

## 3) Требования к аватарам

- Аватар загружается через backend endpoint (`POST /v1/profile/avatar`).
- Файл сохраняется в `DATA_DIR/uploads`.
- В профиль пишется постоянный URL (`/uploads/...`), а не временная `blob:` ссылка.
- При замене аватара старый файл удаляется (best effort).

## 4) Требования к консистентности состояния

- После входа по токену (`/v1/auth/me`) клиент применяет серверный снапшот профиля.
- Первый автоматический `syncProfile` после гидрации должен быть пропущен, чтобы не перетирать серверные данные дефолтами.
- Все изменения профиля и прогресса должны уходить в `syncProfile`.
- При ошибке `syncProfile` UI не должен падать, но должна быть метрика/лог для диагностики.

## 5) Минимальные проверки перед релизом (smoke)

- Регистрация нового пользователя -> выход -> вход: профиль и прогресс сохранены.
- Прохождение диагностики -> выход -> вход: результат диагностики сохранен.
- Смена аватара -> выход -> вход (и другой браузер): аватар отображается.
- Начало и завершение квеста -> выход -> вход: шаги/метрики не сброшены.
- После деплоя backend: старые пользователи доступны, `user not found` не возникает для существующих аккаунтов.

## 6) Инцидент-политика (если обнаружена потеря данных)

- Немедленно проверить:
  - что сервис тот же (`softale-backend`),
  - что диск примонтирован,
  - что `DATA_DIR=/var/data`.
- Проверить наличие свежего бэкапа в `DATA_DIR/backups`.
- При необходимости восстановить `db.json` из последнего валидного бэкапа.
