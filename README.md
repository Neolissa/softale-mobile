# SofTale Mobile MVP

Локальный мобильный MVP на Expo (React Native).

## О проекте

SofTale — мобильная LitRPG-тренировка soft skills в конфликтных сценариях.  
Пользователь проходит сюжетные эпизоды, выбирает реплики в напряженных диалогах, получает последствия выбора, XP и итоговую развязку с поведенческими шкалами.

Ключевая идея продукта:
- обучение коммуникации через игровую драму, а не через сухие тесты;
- короткие, понятные шаги с реальными жизненными конфликтами;
- персонализация через профиль конфликтного стиля;
- удержание интереса за счет ветвления, концовок, достижений и прогресса.

## Текущий стек

- `React Native` + `Expo`
- `TypeScript`
- `AsyncStorage` для локального аккаунта, прогресса и аналитики
- `@expo/vector-icons` (`Feather`, `MaterialCommunityIcons`) + `Emoji` для визуальной системы

## Структура MVP

Сейчас основная логика сосредоточена в одном файле:
- `App.tsx` — UI, состояние, контент, LitRPG-движок, локальная аналитика, профиль.

Публичные сущности в проекте:
- **Сюжеты/курсы**: `forest`, `romance`, `slytherin`, `boss`, `narcissist` + курсовые кампании.
- **Контент шага (источник истины)**: `stepLibrary.ts` (`stepLibraryByCampaign`) — сцена, оппонент, реплика, вопрос, подсказка и пять ответов на одном индексе; реакции NPC собираются офлайн: `npm run build:npc-reactions` → `content/npc-reactions/all.json`.
- **Форматы шагов**: `single`, `multiple`, `builder`.
- **Режимы**: `story` и `course`.
- **Сложности**: `5 / 10 / 15 / 25` вопросов.
- **Финалы**: 4 уникальные развязки на кампанию (`order`, `harmony`, `boundary`, `breakthrough`).

## Основные пользовательские потоки

- **Онбординг/аккаунт**: локальная регистрация и вход (email/password).
- **Диагностика**: 10 ситуаций -> профиль конфликтного стиля + рекомендации.
- **Квест/курс**:
  - сцена -> вопрос -> выбор реплики,
  - проверка ответа,
  - награда/ошибка/штраф,
  - движение по эпизодам и веткам,
  - финал с концовкой, шкалами и достижением.
- **Профиль**: прогресс, достижения, конфликтный стиль.
- **Админка**: локальная аналитика по событиям пользователя.

## Что уже реализовано

- Экран `Карта` (главный цикл дня, вход в квест).
- Экран `Квест` (сюжетные шаги, варианты ответов, XP, ошибки, штрафы, финал).
- Экран `Ивент` (сезонный прогресс и вступление в событие).
- Экран `AI` (обратная связь с уточняющими вопросами).
- Экран `Профиль` (ранг и коллекция артефактов).
- Локальная регистрация/логин с сохранением профиля пользователя.
- Диагностика конфликтного стиля (10 вопросов + выдача результата).
- Локальная аналитика пользовательских событий и прогресса.
- Нижняя таб-навигация без внешних библиотек.
- Единая визуальная система изображений (иконки, emoji, fallback-заглушка).

## Визуальная система (изображения)

### Правило источников

- `Feather` (`@expo/vector-icons`) -> навигация и функциональные иконки.
- `Emoji` -> сюжетные маркеры квестов и артефактов.
- `MaterialCommunityIcons` -> карточные иллюстрации и состояния.
- `Fallback` -> единая заглушка, если у сущности нет изображения.
- Крупная иллюстрация шага в квесте/курсе -> только `Emoji`, подобранный по смыслу реплики оппонента и контекста сцены.

Важно: один тип контента использует только один источник (без смешивания).

### Базовые размеры

- `tab icon`: 18x18
- `inline icon`: 16x16
- `chip icon`: 14x14
- `card leading icon`: 20x20
- `card illustration`: 64x64
- `profile avatar`: 72x72
- `fallback`: 96x96

## LitRPG-контракт квестов

Все сюжеты и курсы работают как LitRPG-кампании на одном движке.

- Каждый квест поддерживает до `25` поворотов.
- Каждый поворот дает выбор из `5` тактик (ветки): `strategist`, `empath`, `boundary`, `challenger`, `architect`.
- Выбор влияет на:
  - реактивный текст сцены (`sceneByBranch`),
  - доминирующую ветку прохождения,
  - финальную концовку.
- Концовки: `4` уникальные развязки на кампанию (`order`, `harmony`, `boundary`, `breakthrough`).
- За каждую концовку открываются:
  - `ending:<campaign>:<endingRoute>`
  - `achievement:<campaign>:<endingRoute>`

### Soft skills фокус

Каждая ветка тренирует свой набор навыков:

- `strategist` -> структура переговоров и ясность решений
- `empath` -> деэскалация и эмоциональная валидация
- `boundary` -> ассертивность и удержание границ
- `challenger` -> инициативность и поведенческий разворот конфликта
- `architect` -> системные договоренности и лидерская коммуникация

## Правила написания квестов (Detroit-подача)

Эти правила обязательны для всех новых и переписанных квестов.

1. Реплика вместо ярлыка:
   - Пользователь видит только живые варианты фраз.
   - Запрещены подписи в духе `Стратег`, `Эмпат`, `Архитектор` в UI-выборе.

2. Сцена без служебных меток:
   - В пользовательском тексте не использовать `Поворот 1`, `Шаг 3`, `ветка X`.
   - Нумерация остается только в системном прогрессе интерфейса.

3. Формула драматического шага:
   - Ставка (что можно потерять),
   - Трение (кто и почему давит),
   - Выбор (5 реплик с разной тактикой),
   - Последствие (тон сцены меняется сразу).

4. Реплики короткие и контрастные:
   - 1-2 фразы, читабельно с мобильного экрана.
   - Каждый вариант должен звучать как реальный ответ в диалоге.

5. Педагогика внутри сюжета:
   - Не объяснять терминологией, а показывать эффект выбора.
   - После шага — смысловая обратная связь по soft skills без назидания.

6. Запрет на техтекст в продукте:
   - Никаких внутренних комментариев, меток экспериментов, ID-шников и служебных формулировок в пользовательских карточках.

7. Реплика оппонента должна быть ясной и острой:
   - В каждой сцене обязателен понятный упрек/претензия от оппонента в прямой речи.
   - Пользователь должен за 1-2 секунды понять: в чем обвиняют, что на кону, почему это неприятно.
   - Допускается колюще-бодрящий тон, но без оскорблений личности и токсичной романтизации.

8. Варианты ответа не должны повторяться от шага к шагу:
   - На каждом вопросе — новый набор реплик, привязанный к текущей сцене.
   - Нельзя использовать один и тот же универсальный список ответов для всей кампании.

9. В каждом шаге должны быть «ловушки»:
   - Помимо конструктивных тактик обязательно добавлять 1-2 ошибочных варианта (токсичность, уход, пассивная агрессия, ультиматум).
   - Ошибочные варианты должны выглядеть правдоподобно, а не карикатурно, чтобы выбор был неочевидным.

10. У шага должен быть один лучший ответ:
   - `single-choice` шаг оценивается по лучшему ходу в контексте сцены (не `acceptAny`).
   - Выбор влияет на ветку и концовку только если это тактически валидный ответ.

11. Сначала история, потом вопрос:
   - Для каждого квеста сначала задается сюжетная арка (завязка -> обострение -> перелом -> финальный рывок -> развязка).
   - Каждый вопрос формулируется как продолжение текущей сюжетной стадии, а не как отдельный абстрактный кейс.

12. Вопрос и ответы не дублируют друг друга:
   - Формулировка вопроса дается отдельной строкой в `instruction`.
   - Варианты ответа не повторяют дословно текст конфликта, а показывают разные поведенческие стратегии.

13. Финал обязан показывать результат по шкалам:
   - После квеста показываются шкалы поведения по веткам (структура, эмпатия, границы, решительность, системность).
   - Обязательно отображать концовку, итог по XP, достижение и краткий смысловой вывод.

## Система вычитки и написания новых квестов

Этот блок фиксирует твои правки как обязательный стандарт качества для всех новых квестов.

### 1) Каркас написания (сначала сюжет, потом механика)

1. Задать арку квеста из 5 стадий:
   - завязка,
   - обострение,
   - перелом,
   - финальный рывок,
   - развязка.
2. На каждую стадию написать отдельные сцены, где повышается ставка.
3. Только после сцен формулировать вопросы и варианты ответов.

### 2) Правила формулировки сцены

- Сцена пишется живым языком, без техметок.
- В сцене всегда есть понятный источник давления (кто и чем давит).
- Реплика оппонента дается в прямой речи и с персонажем:
  - пример: `Настя переводит на тебя внимание: "Хватит тормозить процесс своим занудством"`.
- Запрещены служебные формулы вроде `Фокус эпизода`, `ветка X`, `тип шага`.

### 3) Правила формулировки вопроса

- Вопрос короткий (1 строка), понятный с первого чтения.
- Вопрос не повторяет текст сцены и не дублирует цитату оппонента.
- Вопрос связан с текущей стадией арки (не абстрактный, а сюжетный).
- Нельзя использовать один и тот же вопрос подряд в соседних шагах.

### 4) Правила вариантов ответа

- Каждый вариант — отдельная реплика, а не пересказ сцены.
- В шаге должны быть:
  - сильные тактические варианты,
  - 1-2 правдоподобных ошибочных варианта (не карикатурных).
- Реплики не должны повторяться дословно между соседними шагами.
- Тон: взрослый, острый, но без токсичной романтизации.

### 5) Финал квеста (обязательный комплект)

- 4 уникальные развязки на каждый квест.
- Итоговая концовка с коротким нарративом.
- Подсчет результата: XP, ошибки, штрафы.
- Шкалы поведения по веткам.
- Достижение с человекочитаемым названием.

### 6) Чеклист вычитки перед релизом

Перед публикацией каждого квеста пройтись по чеклисту:

- [ ] В сценах нет техтекста и служебных маркеров.
- [ ] Вопросы не дублируют сцену и не повторяются подряд.
- [ ] В каждом шаге есть вариативность и неочевидный выбор.
- [ ] Повторы реплик между соседними шагами удалены.
- [ ] Реплики оппонентов ясные, конкретные, с понятной претензией.
- [ ] Педагогический смысл сохраняется без назидания.
- [ ] Финал содержит 4 развязки, шкалы, достижение и итог по XP.

### 7) Шаблон нового шага (рабочий)

Использовать как референс при написании:

- `scene`: кто давит, чем рискуем, что на кону.
- `opponent line`: короткая прямая речь с персонажем.
- `instruction`: короткий вопрос по текущей стадии арки.
- `options`: 4-5 реплик (сильные + ошибочные).
- `hint`: не ответ, а направление мышления.
- `result`: что изменилось в отношениях/власти/границах после выбора.

## Контент-валидация (обязательная)

Для всех кампаний действует строгий стандарт качества:

- ответы пишутся живым языком, без служебных маркеров и технических хвостов;
- по мере движения к плохим концовкам и к «жести» сцены и ловушки эскалируют;
- **этапы** привязаны к шагу линейно: `этап = floor(индексШага / шаговНаЭтап)` — без дробления этапа эвристикой;
- варианты ответа не должны повторяться между вопросами одной кампании и между разными кампаниями.

Технически в приложении включен runtime-аудит глобальной уникальности вариантов:

- генератор прогоняет все кампании и сложности;
- строки нормализуются и сравниваются на exact-duplicate;
- найденные коллизии логируются как `content-audit` в консоль разработчика.

Если аудит сообщает коллизии, релиз блокируется до устранения дублей в контенте.

Дополнительно:

- `npm run content:audit` — сканирует `scenarioBible.ts`, `questContent.ts`, `stepLibrary.ts`, `reactionPoolsByCampaign.ts`.
- `npm run build:npc-reactions` — пересобирает JSON реакций после правок `stepLibrary` / `reactionPoolsByCampaign`.
- Таблица длинных кампаний и порядок ручного дополнения: `docs/content-audit-long-campaigns.md`.

## Запуск

1. Убедись, что Node.js не ниже `20.19.4` (лучше LTS 22.x).
2. Установи зависимости:
   - `npm install`
3. Запусти проект:
   - `npm run android` (Android эмулятор)
   - `npm run ios` (iOS симулятор)
   - `npm start` (QR + веб-меню Expo)

## Server mode (production-like)

Для подготовки релиза под RuStore используй server mode:

1. Подними backend:
   - `npm --prefix ./backend install`
   - `npm --prefix ./backend run start`
2. В `.env` приложения задай:
   - `EXPO_PUBLIC_AUTH_MODE=server`
   - `EXPO_PUBLIC_ECONOMY_MODE=server`
   - `EXPO_PUBLIC_ECONOMY_API_BASE_URL=http://localhost:3000`

## Production backend (RuStore)

Актуальный публичный backend для релизных сборок:

- `https://softale-mobile.onrender.com`
- health-check: `https://softale-mobile.onrender.com/health`

Важно:

- Этот URL используется в `eas.json` для профиля `rustore`.
- Для store-сборки запрещено использовать `localhost` или `127.0.0.1`.
- Для модерации RuStore auth-smoke должен проходить против этого URL (`register/login/me`).

В этом режиме:
- регистрация/логин идут через backend auth API;
- пользователи и профиль хранятся на сервере;
- экономика и платежные ордера идут через backend endpoints.

## Деплой в RuStore

Минимальный рабочий сценарий публикации:

1. Проверь release-чеклист:
   - `docs/rustore-release-checklist.md`
   - `docs/clean-release-cycle.md`
2. Убедись, что backend доступен и работает в server mode.
3. Собери Android AAB под RuStore:
   - `npm run android:rustore`
4. Забери артефакт `.aab` из EAS Build и прогони smoke на чистом устройстве.
5. Загрузи AAB в кабинет RuStore и заполни карточку приложения.

Примечания:
- В `eas.json` есть отдельный профиль `rustore` (server-mode env + `app-bundle`).
- Для production включен автоинкремент build-номера (`autoIncrement`), чтобы не ловить конфликт `versionCode`.

### Чистый релизный цикл (обязательно)

В проекте закреплен единый релизный стандарт:

- `docs/clean-release-cycle.md`
- `.cursor/rules/release-cycle.md`

Релиз в RuStore выполняется только по этому циклу: security hygiene -> backend readiness -> build/signing -> smoke -> go/no-go.

## Состояние данных

Проект использует локальное хранение (`AsyncStorage`) и не требует сервера для MVP:

- аккаунты и текущий пользователь;
- профиль (XP, прогресс, выбранные режимы, достижения);
- диагностические ответы и результаты;
- лента аналитических событий.

Важно: это MVP-уровень. Для production понадобятся серверная авторизация, безопасное хранение паролей, синхронизация устройств и бэкенд-аналитика.

## Живой просмотр в эмуляторе (WSL-friendly)

Если проект в WSL2, а эмулятор запущен в Android Studio на Windows, используй:

- `npm run preview:android` — Expo Go режим (быстрый просмотр UI)
- `npm run preview:android:dev` — dev client режим (после установки dev build)
- `npm run go` — короткий алиас для `preview:android`

Что делает скрипт:

1. Проверяет, что эмулятор запущен.
2. Настраивает `adb reverse` для порта `8081`.
3. Автоматически пытается открыть Expo Go с правильным URL.
4. Поднимает Metro на `localhost`, чтобы эмулятор видел бандл.

## Примечание по окружению

В текущем окружении наблюдались ошибки загрузки npm tarball (`ENOENT` / corrupted tarball), из-за чего установка зависимостей может прерываться. Обычно помогает:

- обновить Node/npm,
- `npm cache clean --force`,
- удалить `node_modules` и `package-lock.json`,
- повторить `npm install`.
